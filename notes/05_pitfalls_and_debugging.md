# 技术坑点与调试指南

## 1. 六大核心坑点

### 坑点 1: Reward Hacking（奖励欺骗）

**症状**: 奖励分数上升但实际质量下降；突然的奖励尖峰

**示例**:
- 模型生成看起来合理但完全错误的推理链，骗过神经奖励模型
- 模型输出极长文本以最大化与长度相关的奖励
- 模型直接复制搜索结果而不推理
- 重复短语 exploit 奖励模型的偏差

**解决方案**:
1. **用规则化奖励**（最佳防御 — DeepSeek-R1 的做法）
2. **KL 散度惩罚** — 防止策略偏离太远
3. **奖励裁剪/归一化** — 限制奖励到 [-C, C]
4. **动态奖励缩放**（ToolRL）— 平滑缩放更支持学习
5. **Retrieved token masking** — 防止模型记忆环境输出
6. **不要用长度奖励**（ToolRL 发现会损害性能）
7. **混合框架 (HPA)** — PPO + DPO 互相正则化

**调试方法**: 手动检查高奖励样本。如果看起来不好，说明奖励设计有问题。

---

### 坑点 2: Template Collapse（模板坍缩）

**来源**: RAGEN 论文的关键发现

**症状**:
- 对任意**单个**输入，模型的多个采样看起来多样
- 但对**不同**输入，模型使用完全相同的策略
- 输出表面多样，实质单一

**为什么难以发现**:
```
❌ 传统方法: 计算输出熵 → 看起来正常（单输入内多样）
✅ 正确方法: 计算输入-输出互信息 → 发现不同输入的输出高度相似
```

**解决方案**:
- 用互信息代理指标替代熵
- 确保训练数据中初始状态多样
- 增加采样频率
- 中等粒度的交互（不要太粗也不要太细）

---

### 坑点 3: Echo Trap（回声陷阱）

**来源**: RAGEN 论文

**症状**:
- 奖励方差坍缩到接近零
- 所有 G 个采样输出得分几乎相同
- 梯度出现尖峰
- 模型陷入重复模式

**根本原因**:
当所有采样得分相似时，GRPO 的分母 `std(r)` 接近 0，导致优势估计数值不稳定。

**解决方案**:
```python
# 1. 轨迹过滤: 跳过方差太低的组
if std(rewards) < threshold:
    continue  # 不用这组数据更新

# 2. SNR 自适应过滤 (RAGEN V2)
snr = mean(rewards) / (std(rewards) + eps)
if snr > snr_threshold:
    continue  # 信噪比太高 = 方差太低

# 3. 梯度裁剪
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
```

### 坑点 4: 工具调用格式崩坏

**症状**:
- 模型不再产生有效的工具调用
- 标签变得畸形（如 `<search>` 缺少闭合标签）
- 环境无法解析调用 → 模型得不到反馈 → 永远学不会

**根本原因**:
RL 优化将策略推离结构化输出模式。格式只是 token 序列，没有硬约束。

**解决方案**:

```
方案 1: 格式重罚
  R1-Searcher: 格式违规 -2 分（重罚）

方案 2: 冷启动 SFT
  先在正确格式数据上 SFT → 再做 RL

方案 3: 分阶段训练
  Stage 1: 只训格式奖励
  Stage 2: 格式 + 准确性

方案 4: 约束解码
  在生成时强制模式匹配（如强制闭合标签）
```

**调试**: 单独追踪格式奖励；检查失败样本的输出。

---

### 坑点 5: 长期信用分配困难

**问题**: 多轮工具使用创建长轨迹。奖励归因到正确的动作非常困难。

```
think → search1 → observe1 → think → search2 → observe2 → answer
                                        ↑
                     这次搜索贡献了多少？很难说
```

**解决方案**:
| 方案 | 做法 | 项目 |
|------|------|------|
| 过程奖励 | 每步工具调用给分 | Agent-R1 |
| 轨迹级优化 | StarPO 框架 | RAGEN |
| GiGPO | Group-in-Group 轨迹优化 | verl-agent |
| Critic | 学习的值函数（需要额外显存） | PPO |

---

### 坑点 6: 冷启动 vs SFT 初始化的选择困难

**矛盾发现**:

| 研究 | 结论 |
|------|------|
| DeepSeek-R1 | R1-Zero (纯 RL) 能工作但有可读性问题；冷启动 SFT + RL 最佳 |
| ToolRL | 冷启动 GRPO (从 base 模型 RL) **优于** SFT 初始化的 GRPO |
| R1-Searcher | 无 SFT 冷启动也能工作（两阶段设计） |
| RAGEN | 在 instruct 模型上 GRPO 可能奖励坍缩 |

**建议**:
- 如果任务简单（格式固定）→ 纯 RL 可行
- 如果任务复杂（多工具协调）→ 冷启动 SFT + RL
- 如果用 instruct 模型 → 考虑 PPO 而非 GRPO
- **不要假设 SFT 初始化一定更好** — 它可能限制探索

## 2. 训练稳定性问题速查表

| 问题 | 症状 | 解决方案 |
|------|------|----------|
| 奖励坍缩 | 奖励平台期或突然下降 | 轨迹过滤、梯度裁剪、降低学习率 |
| 梯度爆炸 | Loss 变 NaN、发散 | 梯度范数裁剪、降低学习率 |
| 奖励震荡 | 奖励剧烈跳动 | 减少 PPO epoch、增大 batch |
| 高 value loss (PPO) | Critic 学不好 | Critic 单独学习率、更多 Critic 训练步 |
| 过度搜索 | Agent 不必要地搜索 (20-28%) | β-GRPO 置信度感知奖励 |
| 搜索不足 | Agent 该搜时不搜 (最高 63%) | 课程学习、从简单到难的查询 |
| 模式坍缩 | 输出变得重复/单一 | 增大 β、提高采样温度、确保奖励有方差 |

## 3. 超参数调优优先级

```
最重要 ─────────────────────────────── 次重要
学习率 > 组大小G > KL系数β > clip比例ε > 采样温度 > PPO epochs
```

**学习率**: 1e-6 ~ 5e-6，越低越稳定
**组大小 G**: 8 是好的起点，16 更稳定但更慢
**KL 系数 β**:
  - 太低 → 策略漂移 → 模式坍缩
  - 太高 → 策略不变 → 学不到东西
  - 有趣发现: ToolRL 发现**完全移除 KL** 在工具任务上收敛更快

## 4. 系统工程坑点

### 变长 rollout 的 batch 处理
```
问题: 不同样本需要不同次数的工具调用
  样本 A: think → search → answer          (3 步)
  样本 B: think → search → think → search → answer  (5 步)

解决:
  1. Step-independent rollout (verl-agent): 每步单独推理
  2. Padding + masking: 补齐短序列，mask 环境 token
  3. 设置最大交互预算 B: 硬限制搜索次数
  4. 异步 rollout: 解耦生成和训练
```

### vLLM 集成的注意事项
```
- 训练和推理用不同的模型副本
- 训练更新后需要同步权重到 vLLM 服务器
- 连续 batching 对变长输入很重要
- PagedAttention 节省 KV cache 显存
```

### 环境交互延迟
```
问题: 搜索引擎调用有网络延迟
解决:
  - 批量并行搜索请求
  - 本地搜索索引（Search-R1 用 Pyserini + Wikipedia）
  - 异步处理 + 队列
```

## 5. 调试 Checklist

当训练不如预期时，按以下顺序检查：

```
□ 1. 检查奖励分布 — 是否有足够方差？是否坍缩？
□ 2. 检查格式合规率 — 模型还在产生有效的工具调用吗？
□ 3. 检查 KL 散度 — 策略偏离参考模型多远了？
□ 4. 手动检查 top-5 高奖励样本 — 质量如何？
□ 5. 手动检查 bottom-5 低奖励样本 — 失败模式是什么？
□ 6. 监控梯度范数 — 是否有尖峰？
□ 7. 检查学习率和 warmup — 是否太激进？
□ 8. 检查 batch 内不同 prompt 的奖励差异 — 避免 template collapse
```
